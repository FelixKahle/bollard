# bollard-num

`bollard-num` is a tiny, `no_std`-friendly set of arithmetic traits that mirror the primitive integer APIs in `core` while staying value-centric. It exists to make generic arithmetic over integers feel exactly like writing against the built-in types, without pulling in a broader numeric abstraction layer and without forcing reference-based method signatures.

The primitives in `core` already provide checked and saturating arithmetic as inherent methods that take and return values. When you write `u32::checked_add`, `i64::saturating_mul`, or `usize::checked_shl`, you work with values, not references. `bollard-num` preserves that contract at the trait level. Traits like `CheckedAdd`, `SaturatingSub`, and `CheckedShl` take `self` and their operands by value, just like the primitives, and they return `Option<Self>` or `Self` accordingly. This makes generic functions over integers read naturally and compile to the same thing you would get if you wrote them directly against `u32`, `i64`, or `usize`. There is no impedance mismatch from borrowing, no extra trait combinators to remember, and no surprises around const propagation or macro hygiene when you want to forward to the inherent methods.

We didn’t use `num-traits` because its design goals are broader than what we need and its method shapes often rely on references. In code where all numeric types are `Copy`, value-based APIs remove borrow inference from the equation and align with operator traits like `Add<Self, Output = Self>` that are already value-oriented. By keeping the signatures value-based, bounds like `T: CheckedAdd + CheckedShl` compose cleanly with `core::ops` traits, let the compiler inline straight through to the intrinsic implementations, and keep generic arithmetic ergonomic in both runtime and const contexts. The result is a minimal surface area that is easy to read, easy for the optimizer to flatten, and faithful to how the primitives behave.

The crate is intentionally small. It provides checked arithmetic traits for addition, subtraction, multiplication, division, remainder, negation, and shifts, as well as saturating arithmetic traits for addition, subtraction, multiplication, and negation. Each trait is implemented for the built-in signed and unsigned integer types, delegating directly to the corresponding inherent methods in `core`. Shift traits follow the primitive conventions by taking a `u32` shift amount, and saturating negation is only available for signed integers, mirroring the primitives. All implementations are `#[inline(always)]` to make the traits a zero-cost shim over the underlying operations.

If you want a broad set of numeric abstractions covering many domains, `num-traits` is great. If you want lightweight, by-value arithmetic traits that behave exactly like the intrinsic integer APIs and avoid reference-based method signatures, `bollard-num` gives you just that, with no `std` dependency and no surprises.
